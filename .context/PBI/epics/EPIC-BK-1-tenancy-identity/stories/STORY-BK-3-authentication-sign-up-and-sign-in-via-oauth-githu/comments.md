### Q1 — Canonical post-OAuth redirect: `/home` or `/projects`?

***CONFIRMED: The story is wrong. Correct redirect is `/onboarding` for first-time users, `/projects` for returning users.***

Reasoning from the product model:

- `/home` does not exist as a route. `app/page.tsx` redirects root `/` to `/login`. There is no "home" page — the app's entry point after authentication is either the onboarding flow or the projects list.
- The IQL model is workspace-gated. A user without a workspace cannot use the product. First-time OAuth users have no workspace yet, so they must land on `/onboarding` to create one — identical to the magic-link first-login flow.
- Returning OAuth users already have a workspace → `/projects` directly.

***Updated post-OAuth redirect decision:***

| ***User type**** | ****First OAuth login**** | ****Returning OAuth login*** |
| --- | --- | --- |
| Has workspace | — | `/projects` |
| No workspace | `/onboarding` | — |

AC-1 and AC-2 must be updated to reflect this. The `/home` reference in the workflow is a copy error from an earlier spec draft — discard it.

---

### Q2 — Observable evidence of workspace creation

***CONFIRMED: Two observable signals, in priority order:***

1. ***Primary (UI)***: User lands on `/onboarding`, fills the workspace form, submits, and is then redirected to `/projects` — where the workspace name appears in the app sidebar/header. This is the testable end state: user is on `/projects` with a visible workspace.
2. ***Secondary (URL)***: The URL path `/projects` after onboarding completion is sufficient as a test assertion — does not require DB inspection.

For the integration outline (outline 18), a DB assertion via `workspace_members` table is acceptable to verify the row was created. QA can use the staging DB MCP for that specific outline.

***Do NOT*** require direct `auth.users` inspection as an assertion — that table is Supabase-managed and not directly queryable from the app layer.

---

### Q3 — EMAIL_EXISTS user-facing behavior

***CONFIRMED: URL error param + Sonner toast. No page redirect to an error page.***

Pattern must be consistent with existing error handling in `app/auth/callback/route.ts`:

- Browser redirect: `/login?error=email_exists`
- UI: Sonner toast (already in the stack, `sonner 2.0`) renders from the `error` query param on the login page
- Message: **"An account with this email already exists via a different provider. Contact support to link accounts."**
- No raw HTTP 4xx to the browser — the browser always receives a redirect (3xx); the error is in the URL param

HTTP status from the server to the browser: ***302 redirect*** to `/login?error=email_exists`. The error handling is client-side (toast on mount if `error` param present).

---

### Q5 — Workspace bootstrap failure behavior

***CONFIRMED: Option (b) — persist session, redirect to `/onboarding`. Do NOT rollback.***

Rationale:

- OAuth code exchange is an irreversible Supabase Auth operation. Rolling back the session after a successful code exchange would mean the user is authenticated with the provider but not in our system — creating a worse ghost-user scenario than the bootstrap failure itself.
- The onboarding page already handles the "authenticated but no workspace" state — it shows the workspace creation form. If bootstrap fails server-side, redirect the authenticated user to `/onboarding` and let them complete workspace creation there.
- Log the failure server-side with the user ID and error details for ops visibility.

***Decision***: POST OAuth success → attempt workspace auto-creation → if fails → redirect to `/onboarding` (do not redirect to `/login`, do not rollback session).

### Q4 — OAuth callback route strategy: extend or dedicated?

***CONFIRMED: Extend the existing `app/auth/callback/route.ts`. Single route, branch on `state` param.***

Technical rationale:

1. ***Supabase PKCE handles both flows with the same method.*** `supabase.auth.exchangeCodeForSession(code)` works for both magic-link OTP and OAuth PKCE — Supabase's SDK handles the distinction internally. We do not need to call a different method.

1. ***Single Supabase redirect URI is simpler.*** Having one redirect URI (`${NEXT*PUBLIC*APP_URL}/auth/callback`) registered in both the GitHub OAuth app and Google OAuth app — same as what magic-link already uses — reduces configuration surface. A dedicated `/auth/oauth/callback` route would require registering two additional URIs in Supabase dashboard AND in both provider OAuth apps. More config = more misconfiguration risk.

1. ***State parameter is the discriminator.*** OAuth callbacks always carry `?state=...`. Magic-link callbacks never do. The branch is clean:

```
export async function GET(request: NextRequest) {
  const { searchParams, origin } = new URL(request.url);
  const code = searchParams.get('code');
  const state = searchParams.get('state'); // present = OAuth PKCE, absent = magic-link OTP
  const next = searchParams.get('next') ?? '/projects';

  // Open-redirect guard (unchanged)
  const safeNext = next.startsWith('/') && !next.startsWith('//') ? next : '/projects';

  if (!code) {
    return NextResponse.redirect(`${origin}/login?error=missing_code`);
  }

  // State validation for OAuth (skip for magic-link — no state param)
  if (state) {
    const storedState = // retrieve from httpOnly cookie set at OAuth initiation
    if (state !== storedState) {
      return NextResponse.redirect(`${origin}/login?error=oauth*state*mismatch`);
    }
  }

  const supabase = await createClient();
  const { error } = await supabase.auth.exchangeCodeForSession(code);

  if (error) {
    // Map known Supabase error messages to our error codes
    if (error.message?.includes('email already')) {
      return NextResponse.redirect(`${origin}/login?error=email_exists`);
    }
    const reason = encodeURIComponent(error.message);
    return NextResponse.redirect(`${origin}/login?error=otp*exchange*failed&reason=${reason}`);
  }

  // Post-auth redirect: check workspace membership
  const { data: { user } } = await supabase.auth.getUser();
  if (user) {
    const { data: memberships } = await supabase
      .from('workspace*members').select('workspace*id')
      .eq('user_id', user.id).eq('status', 'active').limit(1);
    if (!memberships || memberships.length === 0) {
      return NextResponse.redirect(`${origin}/onboarding`);
    }
  }

  return NextResponse.redirect(`${origin}${safeNext}`);
}
```

1. ***State token storage***: use an httpOnly cookie set at OAuth initiation time (in the `signInWithOAuth` client call), validated and cleared on callback. Supabase's PKCE helper (`supabase.auth.signInWithOAuth` with `flowType: 'pkce'`) handles state generation natively — we should use that rather than rolling our own.

### Q3 — EMAIL_EXISTS UI treatment (Design specification)

***CONFIRMED: Sonner toast, destructive variant. NOT an inline form error, NOT a full error page.***

Design rationale:

- The login screen uses `magic-link-form.tsx` as its primary interactive surface. The OAuth buttons sit below a divider (`OR`), outside the form. An inline form error would be misleading because no form field caused the error.
- A full redirect to an error page breaks the flow and is disproportionate for a recoverable scenario (user can try magic-link instead).
- Sonner (already in stack at `sonner 2.0`) is the correct component — it renders above all content without disrupting layout.

***Exact toast specification:***

| ***Attribute**** | ****Value*** |
| --- | --- |
| Variant | `toast.error(...)` (destructive — maps to `--fail #e5484d` background tint) |
| Title | `"Account already exists"` |
| Description | `"This email is registered via a different provider. Contact support to link accounts."` |
| Duration | `6000ms` (error toasts stay longer than info toasts) |
| Position | `top-center` (consistent with other auth-flow feedback in the login screen) |
| Dismiss | Manual X + auto-dismiss at 6s |

***Additionally***: the OAuth button that was clicked should briefly show an error state — `data-variant="danger"` (outlined red, per DESIGN.md §6 Button vocabulary) for 2–3s, then revert to disabled. This gives spatial context ("that button caused the error") even as the toast is read.

### Q5 — Workspace bootstrap failure UI (Design specification)

***CONFIRMED: Inline error state within the onboarding flow. NOT a full error page. NOT a toast.***

Design rationale:

- The onboarding screen (`onboarding/page.tsx`) is a focused, single-purpose surface. If workspace creation fails at the RPC level, the error is scoped to that form action — an inline Card with error state is more contextually appropriate than a toast.
- Toasts are for transient background events. A workspace creation failure is a foreground, blocking event — the user cannot proceed without resolving it.

***Inline error card specification:***

| ***Attribute**** | ****Value*** |

| Message | `"We couldn't create your workspace. Please try again."` |
| CTA | Primary button — `"Retry"` (accent fill, `--accent #d9543f`) |
| Placement | Below the workspace slug input field, above the submit button |
| Behavior | Error card replaces the form's success state; retry re-submits the creation RPC |

***The submit button*** during the retry state uses a loading spinner (`data-variant="primary"` with `disabled` + spinner icon) — consistent with existing button patterns in the design system.
